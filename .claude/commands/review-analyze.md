# /review-analyze - 앱 리뷰 페인포인트 분석

특정 앱의 리뷰를 심층 분석하여 페인포인트와 프로덕트 기회를 발굴합니다.

## 입력

- `$ARGUMENTS`: 앱 이름 또는 Play Store/App Store URL

## 실행 순서

### Step 1: 리뷰 수집

```bash
node scripts/collect-reviews.js "$ARGUMENTS"
```

수집 결과 JSON 파일이 `generated/review-analysis/{slug}-{날짜}.json`에 저장됩니다.

### Step 2: 데이터 읽기

생성된 JSON 파일을 읽습니다. JSON 구조:
- `platforms[].meta`: 앱 기본 정보 (이름, 평점, 리뷰수, 카테고리, 개발사)
- `platforms[].reviews`: 수집된 리뷰 (recent, helpful_positive, helpful_negative)
- `platforms[].analysis`: 사전 분석 결과 (painPoints, featureRequests, competitorMentions, sentimentTimeline)

### Step 3: 커뮤니티/영상 리서치

앱스토어 리뷰만으로는 "왜 쓰는지", "헤비 유저의 실제 사용 맥락"이 보이지 않는다.
리뷰 데이터의 한계를 보완하기 위해 아래 소스를 추가 조사한다.

#### 3-1. Reddit 리서치 (WebSearch)

앱 이름과 카테고리 키워드로 Reddit을 검색한다. 병렬로 2-3개 검색 실행.

검색 쿼리 예시:
- `site:reddit.com "{앱이름}" app review`
- `site:reddit.com "{앱 카테고리 키워드}" app recommendation` (예: "random voice chat app", "language exchange app")

검색 결과에서 유의미한 스레드가 있으면 WebFetch로 본문을 읽는다.

**추출 포인트:**
- 사용자가 이 앱을 쓰는 **실제 동기/시나리오** (언어학습? 외로움? 데이팅?)
- **대안으로 언급되는 앱**과 그 이유
- 헤비 유저의 **사용 패턴과 불만** (리뷰보다 솔직한 맥락)
- 이탈한 사용자가 **떠난 이유**

#### 3-2. YouTube 리서치 (WebSearch)

앱 관련 리뷰/사용 영상을 검색한다.

검색 쿼리 예시:
- `site:youtube.com "{앱이름}" review`
- `site:youtube.com "{앱이름}" 후기` (한국어)

검색 결과 상위 3-5개 영상의 제목, 조회수, 댓글 수를 기록한다.
유의미한 영상이 있으면 WebFetch로 영상 페이지의 댓글/설명을 확인한다.

**추출 포인트:**
- 영상에서 보이는 **실제 사용 장면** (어떤 순간이 콘텐츠가 되는가 = 핵심 가치)
- 댓글에서의 **사용자 반응과 대안 추천**
- 조회수가 높은 영상의 **앵글** (재미? 유용함? 감동?)

#### 3-3. 한국 커뮤니티 리서치 (선택)

앱이 한국 개발사이거나 한국 사용자가 많은 경우:
- `site:dcinside.com OR site:everytime.kr "{앱이름}"` 또는 관련 키워드

**주의사항:**
- 리서치 결과가 빈약하면 (검색 결과 <3건) 해당 섹션은 "데이터 부족"으로 표기하고 넘어간다
- 리서치에 과도한 시간을 쓰지 않는다. 스텝 전체를 5-8개 검색 이내로 마무리한다
- 핵심은 **리뷰에서 안 보이는 맥락**을 보완하는 것이지, 모든 정보를 수집하는 것이 아니다

### Step 4: 인사이트 리포트 생성

JSON 데이터를 기반으로 아래 템플릿에 맞춰 마크다운 리포트를 작성합니다.

**핵심 원칙:**
1. **모든 내용은 한글로 작성** - 리뷰 인용문은 원문 유지하되 핵심 해석은 한글로
2. **"왜 쓰는지" 중심 분석** - 단순 불만 나열이 아닌, 사용자 동기와 맥락을 이해한 기회 도출
3. **데이터 기반** - 모든 주장에 리뷰/커뮤니티/영상 인용과 수치 근거 첨부
4. **1인 개발자 관점** - 기회 영역은 솔로 개발자가 접근 가능한 범위로
5. **비현실적 제안 금지** - "서버 비용이 드는 걸 무료로 해라" 같은 소비자 희망사항은 인사이트가 아니다. 비용 구조를 인식한 현실적 기회만 제시

## 2. 페인포인트 분석

### 2.1 불만 카테고리 요약

| 카테고리 | 빈도 | 비율 | 심각도 | 대표 키워드 |
|----------|------|------|--------|------------|
| {theme} | {count}건 | {percentage}% | {🔴높음/🟡중간/🟢낮음} | {keywords} |

심각도 판단 기준:
- 🔴 높음: 앱 사용 자체를 방해하는 문제 (crash, login 실패 등)
- 🟡 중간: 사용에는 지장 없지만 불만이 큰 문제 (광고, 가격 등)
- 🟢 낮음: 개선 희망 수준의 문제 (UI 개선, 기능 추가 등)

### 2.2 상세 분석

**대표 리뷰:**
> "{review text}" — ⭐{score}, 👍 {thumbsUp}

> "{review text}" — ⭐{score}, 👍 {thumbsUp}

> "{review text}" — ⭐{score}, 👍 {thumbsUp}

**기회 영역**: [이 불만을 해결할 수 있는 프로덕트/기능 아이디어 1-2문장]

(상위 5개 카테고리까지 반복)

### 6.1 Reddit/커뮤니티

| 소스 | 핵심 내용 | URL |
|------|----------|-----|
| {subreddit/커뮤니티} | {1-2문장 요약} | {link} |

**대안 앱 언급**: [사용자들이 비교하거나 추천하는 앱과 그 이유]

**이탈 사유**: [떠난 사용자들의 실제 이유 — 리뷰 불만과 다른 점이 있다면 언급]

(검색 결과 부족 시 "커뮤니티 데이터 부족 — {검색 키워드}로 유의미한 스레드 미발견" 표기)

## 7. 기회 영역

리뷰 + 커뮤니티 + 영상 데이터를 종합하여 1인 개발자가 접근 가능한 기회를 도출합니다.
기회 영역은 반드시 **리서치에서 발견한 구체적 맥락**에 기반해야 한다. "무료로 해라" 식의 비현실적 제안은 제외.

### 기회 1: {opportunity title}

- **근거**: {어떤 페인포인트/기능요청 데이터에서 도출했는지}
- **아이디어**: {구체적인 프로덕트/기능 아이디어}
- **접근 전략**: {MVP 범위와 차별화 전략}
- **난이도**: ⭐/⭐⭐/⭐⭐⭐

(2-3개 기회 제시)

## 8. 종합 의견

[리뷰 + 커뮤니티 + 영상 데이터를 종합하여 3-5문장으로 핵심 기회 정리]
- 가장 큰 기회는 무엇인가
- 이 앱의 사용자들이 가장 절실하게 원하는 것은 무엇인가
- 1인 개발자가 당장 시작할 수 있는 것은 무엇인가

## 플랫폼별 처리

- URL 입력 → 해당 플랫폼만 분석
- 앱 이름 입력 → Play Store + App Store 양쪽 분석
- 양쪽 데이터가 있으면 통합 분석 (플랫폼별 차이점도 언급)

---

## 에러 처리

- 수집 스크립트 실패 → 에러 메시지 출력하고 중단
- 특정 플랫폼 실패 → 나머지 플랫폼 데이터로 분석 진행
- 리뷰 수가 너무 적으면 (<5개) → "리뷰 수가 부족하여 분석 신뢰도가 낮을 수 있습니다" 경고 추가
