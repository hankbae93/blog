## Top 3 Ideas

### 1. CodeDebt Journal

**한 줄 요약:** AI가 만든 코드의 "부채"를 추적하고, 수동 개입이 필요한 지점을 사전에 알려주는 개발자 도구

**영감 출처:**
- [GeekNews] "2년간의 vibecoding을 끝내고 수동 코딩 복귀" (일관성/구조적 완전성 문제)
- [Dev.to] "We're Creating a Knowledge Collapse and No One's Talking About It"
- [GeekNews] Karpathy의 80% 에이전트 코딩 전환

**Kill Zone 체크:** ✅ 통과
- GitHub Copilot, Cursor 등은 "코드 생성"에 집중
- "AI 생성 코드의 품질 추적/부채 관리"는 아직 빅테크가 안 건드림
- 기존 코드 품질 도구(SonarQube 등)는 AI 컨텍스트를 모름

**차별화 전략:** Niche + Integration
- AI 코딩 도구 사용자만을 위한 틈새 시장
- 기존 IDE/에이전트와 연동하여 "AI가 만든 코드" 태깅

**왜 지금?:**
- Vibecoding이 2년차에 접어들며 "부채 청산" 시점 도래
- AI 코딩 도입률이 임계점(Karpathy 80%)을 넘어섬
- Knowledge Collapse 담론이 표면화되기 시작

**타겟 사용자:**
AI 코딩 도구를 1년 이상 사용해온 시니어 개발자. 코드베이스가 커지면서 "어디가 AI가 만든 부분이고, 왜 이렇게 되어있는지" 파악이 어려워진 사람.

**해결하는 문제:**
AI가 생성한 코드는 즉시 작동하지만, 시간이 지나면 왜 그렇게 짜여졌는지 이해하기 어렵다. 수동으로 리팩토링해야 할 부분을 사전에 식별하지 못하면 기술 부채가 폭발한다.

**MVP 핵심 기능:** (max 3)
1. **AI Origin Tracker**: 커밋/파일 단위로 "AI 생성 비율" 자동 태깅
2. **Debt Score Dashboard**: 시간에 따른 AI 코드 부채 누적 시각화
3. **Refactor Alert**: 특정 임계치 도달 시 "수동 검토 필요" 알림

### 본질 평가
| 질문 | 판정 | 근거 |
|------|------|------|
| 전제 파괴 | ✅ | "AI 코드는 빠르니까 좋다"는 전제를 깬다. 속도의 대가(부채)를 가시화 |
| 방치 이유 | ✅ | AI 코딩이 아직 2-3년차. 부채가 터지기 전이라 아무도 안 다룸 |
| 비합리적 행동 | ✅ | 개발자들이 AI 코드를 그대로 두고 "나중에 보자" 하며 방치 중 |
| 기능-가설 연결 | ✅ | 3가지 기능 모두 "AI 부채 추적"이라는 가설과 직결 |
| 진정성 | ⚠️ | vibecoding 경험자가 만들어야 설득력. 직접 경험 필요 |

**종합 판정:** ✅ 추천
**핵심 피드백:** "AI 코드 생성"이 아닌 "AI 코드의 사후 관리"라는 관점이 날카롭다. 다만, 본인이 vibecoding으로 고통받은 경험이 없다면 문제의 디테일을 놓칠 수 있다. 직접 AI 코딩 2년 경험자를 인터뷰하거나, 본인이 3개월간 100% AI 코딩을 해본 후 만들 것.

### 2. MCP Bridge

**한 줄 요약:** AI 코드 에디터마다 다른 MCP 서버 설정을 한 곳에서 관리하고, 팀 전체에 동기화하는 도구

**영감 출처:**
- [GeekNews] "MCP Server 설정 파일 변환기" (여러 에디터 간 설정 변환 필요)
- [Product Hunt] Zenvault - "One CLI to onboard any project instantly"
- [GitHub] pi-mono - "AI agent toolkit: coding agent CLI, unified LLM API"

**Kill Zone 체크:** ✅ 통과
- Anthropic이 MCP 스펙을 만들었지만 "설정 관리"는 안 함
- 각 에디터(Cursor, Claude Code 등)가 자체 설정 형식 고수
- 통합 관리 도구는 빅테크의 관심 밖

**차별화 전략:** Integration + Local
- 파편화된 MCP 생태계의 "글루" 역할
- 한국 개발자 커뮤니티 먼저 공략 (GeekNews에서 화제)

**왜 지금?:**
- MCP가 2024년 말 공개 후 급속 확산
- AI 에디터 종류가 폭발적 증가 (Cursor, Windsurf, Claude Code 등)
- 팀 단위 AI 도구 도입이 본격화되면서 설정 동기화 문제 대두

**타겟 사용자:**
5인 이상 개발팀의 테크리드. 팀원마다 다른 AI 에디터를 쓰는데, MCP 서버 설정을 매번 수동으로 맞춰야 하는 상황.

**해결하는 문제:**
MCP 서버(DB 연결, API 키 등)를 팀원 각자가 각기 다른 형식으로 설정해야 한다. 누군가 설정을 바꾸면 전파가 안 되고, 보안 민감 정보가 개인 파일에 흩어져 관리 불가.

**MVP 핵심 기능:** (max 3)
1. **Config Unifier**: 여러 에디터 설정을 하나의 표준 형식으로 변환
2. **Team Sync**: 팀 공용 MCP 설정을 중앙에서 관리, 로컬 동기화
3. **Secret Vault**: API 키 등 민감 정보 분리 저장 (HashiCorp Vault 연동)

### 본질 평가
| 질문 | 판정 | 근거 |
|------|------|------|
| 전제 파괴 | ⚠️ | "설정은 개인이 각자 관리"라는 전제를 깨지만, 혁신적이진 않음 |
| 방치 이유 | ✅ | MCP가 너무 새로움(2024 말). 아직 표준 도구가 나올 시간 부족 |
| 비합리적 행동 | ✅ | 팀원들이 Slack으로 설정 파일 공유, 수동 복붙 중 |
| 기능-가설 연결 | ✅ | 3가지 기능이 "MCP 설정 파편화" 문제와 직결 |
| 진정성 | ⚠️ | 팀 규모 개발 경험이 없으면 문제의 급박함을 모를 수 있음 |

**종합 판정:** ⚠️ 보완 필요
**핵심 피드백:** 문제는 실재하지만 "본질"이라기보다 "불편함"에 가깝다. 더 깊이 파보면, 진짜 문제는 "AI 도구 도입 시 팀 전체의 워크플로우가 깨진다"일 수 있다. 설정 동기화는 그 증상 중 하나일 뿐. MCP 설정뿐 아니라 "AI 도구 팀 도입"이라는 더 큰 문제로 확장할 수 있는지 고민 필요.

### 3. SRE Copilot for Solo Devs

**한 줄 요약:** 1인 개발자가 서비스 운영에서 겪는 "새벽 3시 장애 대응"을 AI가 1차 분류하고, 정말 긴급할 때만 깨워주는 도구

**영감 출처:**
- [GeekNews] "소프트웨어 엔지니어링의 미래는 SRE다" (코딩 → 운영 전문성 이동)
- [Indie Hackers] $60k/mo 앱 포트폴리오 운영자 (1인이 다중 서비스 운영)
- [TechCrunch] Amazon Go/Fresh 폐쇄 (운영비용 문제로 철수)

**Kill Zone 체크:** ✅ 통과
- PagerDuty, Opsgenie 등은 "팀" 대상
- 1인 개발자용 운영 도구는 시장이 작아 빅테크가 안 들어옴
- 기존 도구는 설정이 복잡하고 가격이 비쌈

**차별화 전략:** Niche + Price
- 1인 개발자/인디해커 전용 (팀 기능 과감히 제거)
- 월 $5-10 저가 모델 (PagerDuty $29/user 대비)

**왜 지금?:**
- AI 코딩으로 "만들기"는 쉬워졌지만 "운영"은 여전히 어려움
- 인디해커 성공 사례 증가로 1인 다중 서비스 운영자 급증
- Cloud 가격 인상으로 비용 효율적 운영 도구 수요 증가

**타겟 사용자:**
$5k-50k MRR 규모의 1인 SaaS 운영자. 낮에는 개발하고, 밤에는 자고 싶은데 서비스 장애가 두려워 알림을 끄지 못하는 사람.

**해결하는 문제:**
1인 개발자는 모든 알림을 직접 받는다. 하지만 대부분은 "지금 당장 안 봐도 되는" 것들. 진짜 긴급한 건 10%도 안 되는데, 그걸 분류하려면 결국 다 확인해야 한다.

**MVP 핵심 기능:** (max 3)
1. **Alert Triage AI**: 로그/메트릭 기반 장애 심각도 자동 분류 (Critical/Warning/Info)
2. **Smart Wake**: Critical만 전화/SMS, 나머지는 조용히 정리
3. **Morning Digest**: 밤새 쌓인 Warning/Info를 아침에 요약 리포트로 전달

### 본질 평가
| 질문 | 판정 | 근거 |
|------|------|------|
| 전제 파괴 | ✅ | "운영은 팀이 해야 한다"는 전제를 깬다. 1인도 잘 수 있다 |
| 방치 이유 | ✅ | 1인 운영자 시장이 작아 도구 회사가 관심 없었음 |
| 비합리적 행동 | ✅ | 인디해커들이 새벽에 수동으로 Slack/Discord 알림 확인 중 |
| 기능-가설 연결 | ✅ | 3가지 기능이 "1인 운영자의 수면권"이라는 가설과 직결 |
| 진정성 | ✅ | 인디해커 본인이 만들면 100% 공감. 1인 서비스 운영 경험 필수 |

**종합 판정:** ✅ 추천
**핵심 피드백:** "SRE의 미래" 담론과 "인디해커 성장"이라는 두 트렌드가 교차하는 지점. 문제가 구체적이고, 타겟이 명확하며, 가격 차별화도 가능하다. 다만, 1인 서비스를 운영해본 경험이 없다면 "진짜 긴급한 장애"와 "무시해도 되는 알림"의 기준을 못 잡을 수 있다. 최소 6개월간 자기 서비스를 운영하며 알림 로그를 모아보고 시작할 것.

## Summary

|------|----------|-----------|-----------|
