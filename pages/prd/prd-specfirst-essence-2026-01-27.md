# PRD: SpecFirst

> AI 코딩 도구의 "앞단"에 명세 레이어를 추가하여, 모호한 프롬프트 문제를 해결하는 구조화된 요구사항 작성 플랫폼

- 작성일: 2026-01-27
- 버전: 2.0
- 상태: Draft
- 트랙: Essence (본질 추구형)

## 1. 개요

### 1.1 문제 정의

**핵심 문제:**
AI 코딩 도구(Cursor, Copilot, Claude Code 등)가 "바로 코드 생성"을 해주면서, 개발자들이 "무엇을 만들지" 명확히 정의하지 않고 "일단 만들고 보자" 패턴을 반복하고 있다.

**누가 겪는 문제인가?**
- Cursor, Copilot 등 AI 코딩 도구를 사용하지만 결과물에 만족하지 못하는 개발자
- Vibecoding(감으로 AI에게 지시하는 코딩)을 시도했다가 좌절한 1인 개발자
- AI 도구를 팀에 도입했지만 코드 품질 일관성에 고민인 테크 리드

**무엇이 문제인가?**
- **구조화 부재**: AI에게 뭘 시켜야 할지 모름 (산발적인 요청)
- **재현 불가능**: AI 결과물이 매번 달라서 일관성 없음
- **명세 작성 장벽**: 명세 쓰는 것이 코딩보다 더 힘들게 느껴짐
- **학습 불가**: 과거 시행착오를 다음에 활용할 수 없음

**왜 문제인가? (전제 파괴 관점)**
- **파괴해야 할 전제**: "AI 코딩은 프롬프트만 잘 쓰면 된다"
- **실제 현실**: Vibecoding 2년 경험이 축적되면서, 단순 프롬프트로는 AI 코딩 품질을 담보할 수 없다는 것이 명확해졌다. 실제 필요한 것은 **구조화된 명세(Specification)**이다.
- **증거**: "2년간 Vibecoding 후 다시 수동 코딩으로 복귀"하는 사례가 개발자 커뮤니티에서 공유되고 있다.

**본질적 통찰:**
> "AI가 빠르면 뭐하나, 명세가 흐리면 결과도 흐리다."

### 1.2 솔루션 요약

**SpecFirst**는 AI 코딩 도구 사용자를 위한 **구조화된 명세 작성 플랫폼**이다.

- **구조화**: 기능/제약/에지케이스를 분리한 템플릿으로 모호함 제거
- **최적화**: Cursor/Copilot에 최적화된 포맷으로 원클릭 변환
- **학습**: 명세 히스토리와 결과물 추적으로 지속적 개선 지원

**핵심 가치 제안:**
"AI에게 잘 시키는 것"이 아니라 "AI에게 시킬 것을 명확히 정의하는 것"이 AI 코딩의 본질이다.

### 1.3 성공 지표 (KPIs)

| 지표 | 정의 | 목표 | 측정 방법 |
|------|------|------|----------|
| 명세 재사용률 | 생성된 명세가 2회 이상 재사용되는 비율 | >60% | 명세 복사/수정 이벤트 추적 |
| AI 코딩 성공률 | 명세 기반 AI 코딩의 첫 시도 성공률 | >70% | 사용자 피드백 (thumbs up/down) |
| WAU (주간 활성 사용자) | 주 1회 이상 명세 생성 또는 수정 | 런치 4주 후 1,000명 | 이벤트 로그 |
| Pro 전환율 | Free -> Pro 전환 비율 | >5% | Stripe 결제 데이터 |

## 2. 타겟 사용자

### 2.1 Persona

**Persona 1: 좌절한 Vibecoder - 민수 (32세, 백엔드 개발자)**

| 항목 | 내용 |
|------|------|
| 배경 | Cursor를 6개월째 사용, 스타트업 재직 |
| 동기 | 반복 작업을 AI로 자동화하고 싶음 |
| Pain Point | 복잡한 기능에서 AI 결과물이 엉망, 프롬프트 수정만 반복 |
| 현재 행동 | "차라리 처음부터 내가 짜는 게 빠르겠다"며 AI 사용 포기 |
| 목표 | AI를 "진짜 생산성 도구"로 활용하고 싶음 |
| 기술 수준 | 중급 (3-5년 경력), Python/TypeScript |

**Persona 2: 사이드 프로젝트 빌더 - 지현 (28세, 1인 개발자)**

| 항목 | 내용 |
|------|------|
| 배경 | 아이디어 많음, 주말마다 사이드 프로젝트 시작 |
| 동기 | 빠르게 프로토타입을 만들어 검증하고 싶음 |
| Pain Point | Copilot으로 시작은 빠르지만, 기능 추가될수록 AI가 맥락 놓침 |
| 현재 행동 | "AI한테 설명하는 게 코딩하는 것보다 힘들다"며 포기 |
| 목표 | 체계적으로 AI에게 명세를 전달하는 방법을 배우고 싶음 |
| 기술 수준 | 초중급 (1-3년 경력), React/Node.js |

**Persona 3: 품질 고민 팀 리드 - 현우 (35세, 스타트업 테크 리드)**

| 항목 | 내용 |
|------|------|
| 배경 | 5인 개발팀 리딩, AI 코딩 도구 도입 주도 |
| 동기 | 팀 전체의 AI 활용 수준을 표준화하고 싶음 |
| Pain Point | 팀원마다 AI 활용 수준이 다름, 코드 품질 들쭉날쭉 |
| 현재 행동 | "AI 코딩 가이드라인"을 만들려 하지만 어디서 시작할지 모름 |
| 목표 | 팀 공용 명세 템플릿과 베스트 프랙티스 확립 |
| 기술 수준 | 고급 (7년+ 경력), 풀스택 |

     |
     v
[습관화]
명세 저장 -> 재사용 -> Pro 업그레이드

### 2.3 사용 시나리오

**시나리오 1: 새 기능 명세 작성 (민수)**

1. 민수가 "결제 취소 API" 기능을 Cursor로 구현하려 함
2. SpecFirst에 접속, "REST API 엔드포인트" 템플릿 선택
3. 가이드 질문에 따라 작성:
   - 입력: 결제 ID, 취소 사유 (선택)
   - 출력: 취소 완료 응답 (취소 ID, 환불 금액, 처리 일시)
   - 에러 케이스: 이미 취소된 결제, 취소 불가 상태, 권한 없음
   - 제약 조건: 결제 후 7일 이내만 취소 가능
4. "Cursor용 포맷으로 변환" 클릭
5. 생성된 구조화된 프롬프트를 Cursor에 붙여넣기
6. 첫 시도에 원하는 코드 생성 성공!
7. 명세를 "결제 시스템" 프로젝트에 저장

**시나리오 2: 이전 명세 재활용 (지현)**

1. 지현이 새 사이드 프로젝트에서 "사용자 인증" 기능 필요
2. 2주 전 작성한 "사용자 인증 API" 명세 검색
3. 명세 복사 후 새 프로젝트에 맞게 수정 (OAuth 추가)
4. 버전 히스토리에 변경 사항 자동 기록
5. 수정된 명세로 AI 코딩 진행

**시나리오 3: 팀 표준화 (현우)**

1. 현우가 팀 공용 템플릿 생성 ("우리 팀 API 명세 표준")
2. 팀원들을 Team 워크스페이스에 초대
3. 팀원들이 표준 템플릿 사용하여 명세 작성
4. 코드 리뷰 시 명세 링크 첨부 -> 의도 파악 용이
5. 팀 전체 AI 코딩 품질 일관성 확보

## 3. 기능 요구사항

### 3.1 MVP 필수 기능 (3개)

**기능 1: 구조화된 명세 템플릿**

| 항목 | 내용 |
|------|------|
| 설명 | 기능/제약/에지케이스를 분리한 입력 폼 제공 |
| 본질적 가치 | 모호함을 구조화로 제거 |
| 입력 | 기능 설명, 입력 파라미터, 출력 형식, 에러 케이스, 제약 조건 |
| 출력 | 구조화된 명세 문서 |
| 가이드 | 각 필드별 예시 + 질문 ("이 기능이 실패하면 어떻게 해야 하나요?") |

**템플릿 종류 (MVP):**
- REST API 엔드포인트
- 함수/메서드
- React 컴포넌트

**수용 기준:**
- [ ] 3종 템플릿 제공
- [ ] 각 템플릿에 가이드 질문 5개 이상
- [ ] 빈 필드 경고 (필수 항목 누락 시)
- [ ] 실시간 미리보기

---

**기능 2: AI 도구 최적화 포맷 변환**

| 항목 | 내용 |
|------|------|
| 설명 | 작성된 명세를 Cursor/Copilot 친화적 프롬프트로 변환 |
| 본질적 가치 | 도구와 도구 사이의 마찰 제거 |
| 지원 포맷 | Cursor (마크다운 + 체크리스트), Copilot (인라인 주석), Claude Code (XML 구조) |
| 기능 | 원클릭 복사, 포맷 선택, 프리뷰 |

**변환 예시:**

```markdown
## 기능: 결제 취소 API

### 입력
- `payment_id` (string, required): 취소할 결제 ID
- `reason` (string, optional): 취소 사유

### 출력
- `cancellation_id`: 취소 처리 ID
- `refund_amount`: 환불 금액
- `processed_at`: 처리 일시 (ISO 8601)

### 에러 케이스
- [ ] 이미 취소된 결제 -> 400 Bad Request
- [ ] 취소 불가 상태 -> 409 Conflict
- [ ] 권한 없음 -> 403 Forbidden

### 제약 조건
- 결제 후 7일 이내만 취소 가능
- 부분 취소 미지원
```

**수용 기준:**
- [ ] 3종 포맷 변환 지원 (Cursor, Copilot, Claude Code)
- [ ] 원클릭 복사 기능
- [ ] 복사 완료 피드백 (토스트)
- [ ] 포맷별 프리뷰 탭

---

**기능 3: 명세 히스토리 및 추적**

| 항목 | 내용 |
|------|------|
| 설명 | 작성한 명세 저장, 버전 관리, 검색 |
| 본질적 가치 | 학습과 개선을 가능하게 함 |
| 저장 | 프로젝트별 그룹핑, 태그 |
| 버전 | 수정 시 자동 버전 생성, 변경 이력 |
| 검색 | 키워드, 태그, 프로젝트별 필터 |

**수용 기준:**
- [ ] 명세 CRUD (생성, 조회, 수정, 삭제)
- [ ] 프로젝트별 그룹핑
- [ ] 버전 히스토리 (최근 10개)
- [ ] 키워드 검색
- [ ] 명세 복제 기능

### 3.2 Phase 2 기능

| 우선순위 | 기능 | 설명 | 가치 |
|---------|------|------|------|
| High | AI 보조 명세 작성 | 자연어 입력 -> 구조화된 명세 초안 자동 생성 | 명세 작성 시간 50% 단축 |
| High | 결과 피드백 연동 | AI 코딩 결과 평가 (성공/실패) -> 명세 개선 제안 | 지속적 학습 루프 |
| Medium | 팀 협업 | 공유 워크스페이스, 템플릿 공유, 코멘트 | 팀 표준화 |
| Medium | VS Code 익스텐션 | 에디터 내에서 명세 작성/조회 | 컨텍스트 스위칭 최소화 |
| Low | 명세-코드 연결 | 명세와 생성된 코드 파일 연결/추적 | 트레이서빌리티 |

### 3.3 Out of Scope (명시적 제외)

| 제외 항목 | 이유 |
|----------|------|
| 코드 생성 기능 | Cursor/Copilot의 영역, 경쟁이 아닌 보완 |
| 코드 리뷰 기능 | 과포화 시장 (GitHub Copilot, CodeRabbit 등) |
| 범용 문서 작성 | Notion의 영역, AI 코딩 특화 유지 |
| 프로젝트 관리 | Linear/Jira의 영역, 명세에 집중 |
| 실시간 협업 편집 | Phase 2 이후 고려, MVP 복잡도 관리 |

);
