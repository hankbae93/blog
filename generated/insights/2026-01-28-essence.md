## Top 3 Ideas

### 1. AgentBreak - "AI가 멈춰야 할 순간"을 알려주는 도구

**한 줄 요약:** AI Agent에게 "여기서 멈추고 사람에게 물어봐"라고 말할 수 있는 경계선 설정 도구

**깨는 전제:**
> "AI Agent는 더 많이 자동화할수록 좋다"

대부분의 Agent 도구는 "더 많은 자율성"을 경쟁한다. 하지만 vibecoding 2년 후 수동 복귀 사례와 "Knowledge Collapse" 경고가 보여주듯, **Agent가 멈춰야 할 지점을 모르는 것**이 진짜 문제다. 80% agentic coding은 100%가 아니다. 그 20%가 왜 남아있는지를 묻는 사람이 없다.

**방치 이유:**
- Agent 시장이 "더 강력한 자동화" 경쟁에 빠져 있음
- "멈춤"을 팔면 기술 후퇴처럼 보임
- Agent 제작자들이 "자율성"을 KPI로 삼아서 멈춤을 연구 안 함
- 사용자도 "더 많이 해줬으면" 하는 욕구가 먼저라 요구하지 않음

**비합리한 행동:**
- 개발자들이 Agent가 만든 코드를 전부 검토하다가 시간을 날림
- Agent가 잘못된 방향으로 20분 달린 후에야 "어 이거 아닌데" 깨달음
- 중요한 결정을 Agent에게 맡겼다가 나중에 전부 수동으로 다시 함
- "일단 Agent한테 시켜놓고 나중에 보자" 하며 기술 부채 누적

**핵심 기능:** (max 3)
1. **Boundary Marker**: 코드/파일/결정에 "여기서 멈추고 물어봐" 태그 설정
2. **Agent Pause Protocol**: Agent가 경계에 도달하면 자동 정지 및 컨텍스트 요약 전달
3. **Decision Audit**: Agent가 스스로 결정한 것 vs 사람이 승인한 것 구분 로그

### 2. LocalFirst Agent - 클라우드 없는 개인 AI 비서

**한 줄 요약:** 인터넷 연결 없이, 내 기기에서만 돌아가는 AI Agent

**깨는 전제:**
> "AI는 클라우드에서 더 강력하다"

Moltbot(로컬 AI 비서)이 460 votes로 2위를 차지한 건 우연이 아니다. Apple Silicon용 음성 AI(mlx-audio)가 GitHub Trending에 오른 것도 같은 맥락. supermemory가 "AI용 Memory API"를 만든 건 **Agent가 클라우드에 의존하면 "기억"을 통제할 수 없다**는 걸 보여준다.

**방치 이유:**
- 빅테크는 클라우드 종속이 비즈니스 모델
- "로컬 AI는 약하다"는 2023년 인식이 아직 남아 있음
- Apple Silicon/NVIDIA GPU의 로컬 AI 성능이 알려지지 않음
- 개인정보 민감 사용자가 목소리를 안 냄 (이미 불편함에 익숙해서)

**비합리한 행동:**
- 민감한 정보를 ChatGPT에 복붙하면서 "어쩔 수 없다" 체념
- 회사에서 금지해서 집에 가서 AI 사용
- VPN 켜고 AI 쓰면서 "보안 됐겠지" 자기 위안
- 로컬 LLM 설치하려다 "너무 복잡해서" 포기

**핵심 기능:** (max 3)
1. **One-Click Local Setup**: Ollama + 최적 모델 원클릭 설치 (M1 Mac 기준)
2. **Memory Vault**: 대화 기록, 문서, 노트를 로컬에만 저장하는 AI 메모리
3. **Offline Mode**: 인터넷 없이도 100% 작동하는 Agent 기능

### 3. SingleFile Agent - 20시간 만에 배포되는 초경량 Agent

**한 줄 요약:** 한 파일에 담기는 Agent를 만들고, 남들도 쓸 수 있게 바로 배포

**깨는 전제:**
> "Agent는 복잡한 시스템이어야 한다"

"One Human + One Agent = Browser in 20K LOC"라는 헤드라인과 "20시간 개발 → 5-figure ARR" 사례가 보여주는 건 명확하다: **Agent는 작을수록 빠르고, 빠를수록 검증이 쉽다**. 하지만 현재 Agent 툴킷(pi-mono, LobeHub 등)은 여전히 "풀스택 인프라"를 요구한다.

**방치 이유:**
- Agent 도구 회사들이 "프레임워크"를 팔고 싶어함
- "단순한 Agent"는 기술력이 안 보여서 VC 투자 유치가 어려움
- 개발자들이 "복잡해야 잘 만든 것" 편견을 가짐
- 인디해커들은 각자 방식으로 해결해서 도구화가 안 됨

**비합리한 행동:**
- Agent 하나 만드는데 boilerplate 설정에 2일 소요
- 검증도 안 된 아이디어에 "제대로 된 아키텍처" 고민
- 다른 사람한테 Agent 공유할 때 "설치 가이드" 문서 작성
- Agent가 잘 작동하는지 보려고 별도 테스트 환경 구축

**핵심 기능:** (max 3)
1. **Single File Template**: Agent 로직이 한 파일에 담기는 템플릿 (Python/TypeScript)
2. **Instant Deploy**: 파일 하나 업로드로 즉시 API 엔드포인트 생성
3. **Share Link**: 만든 Agent를 링크 하나로 다른 사람과 공유

## 본질 관점에서의 최종 조언

**공통된 발견:**
세 아이디어 모두 "더 많이, 더 복잡하게, 더 클라우드로"라는 현재 AI 시장의 방향에 역행한다. 이것이 본질 추구형 관점에서 의미 있는 이유는:

1. **역방향에 진짜 문제가 있다** - 시장이 한 방향으로 달리면, 반대 방향의 고통은 방치된다
2. **전제를 깨야 관점이 차별화된다** - "더 강력한 Agent"가 아닌 "멈출 줄 아는 Agent"
3. **단순함이 본질이다** - 복잡함은 문제 해결이 아니라 문제 회피인 경우가 많다

**선택 기준:**
- AgentBreak: Agentic coding을 6개월 이상 경험하고 "어디서 멈춰야 하는지" 직접 고통받은 사람이라면
- LocalFirst Agent: 규제 산업 종사자이거나 프라이버시에 진심인 사람이라면
- SingleFile Agent: 인디해커로서 "빠른 검증"을 최우선으로 생각하는 사람이라면

**경고:**
세 아이디어 모두 "만드는 사람이 문제를 직접 겪어봤는가"가 성패를 가른다. 특히 AgentBreak은 "Agent를 충분히 써본 사람"이 아니면 "멈춰야 할 지점"의 감을 잡을 수 없다. 트렌드를 역주행하는 것은 용기가 필요하지만, 그 용기는 경험에서 나와야 한다.
