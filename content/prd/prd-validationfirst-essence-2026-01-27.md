# PRD: ValidationFirst

> "I Keep Building Apps Before Validating" - 검증 없이 빌딩하는 악순환을 끊는 아이디어 검증 강제 도구

## 1. 개요

### 1.1 문제 정의

- **누가** 겪는 문제인가?
  - 사이드 프로젝트가 계속 실패하는 인디 해커
  - "다음엔 검증 먼저 하겠다"고 다짐하지만 반복 실패하는 1인 개발자
  - 빌드 인 퍼블릭을 하지만 런칭 후 사용자가 없는 경험을 한 메이커
  - AI 코딩 도구로 빠르게 만들 수 있게 되었지만, 더 빨리 실패하는 개발자

- **무엇이** 문제인가?
  - **아이디어 → 바로 코딩** 패턴이 자동화되어 있음 (검증 단계 스킵)
  - **검증의 정의가 모호함**: "검증이 뭔데?" - 랜딩페이지? 인터뷰? 설문?
  - **검증 귀찮음 >> 코딩 재미**: 개발자는 만드는 것이 즐겁고, 검증은 마케팅 영역으로 느껴져 회피
  - **"이번엔 다를 거야" 착각**: 과거 실패에서 배우지 않고 같은 패턴 반복

- **왜** 문제인가? (전제 파괴 관점)
  - **파괴해야 할 전제:** "일단 만들어봐야 시장 반응을 안다"
  - **실제 현실:** 만들기 전에도 충분한 시장 신호를 얻을 수 있다. 대기자 명단, 사전 판매, 인터뷰, 경쟁사 분석만으로도 Go/No-Go 결정이 가능하다.
  - **비합리적 행동의 구조:**
    1. 아이디어가 떠오름
    2. 흥분 상태에서 바로 `npx create-next-app` 실행
    3. 2-4주 개발
    4. 런칭: "아무도 안 씀"
    5. 좌절 후 새 아이디어로 1번 반복
  - **AI 시대의 악화:** Cursor, Copilot 덕분에 개발 속도가 3배 빨라짐 → "쓸모없는 것을 3배 빠르게 만드는" 상황 발생

### 1.2 솔루션 요약

ValidationFirst는 **코딩 전에 검증을 강제하는 워크플로우 도구**이다.

핵심 철학: **"빌드 버튼을 누르기 전에 검증 완료 체크박스를 통과해야 한다"**

- **Step 1 (강제):** 아이디어 입력 → 검증 체크리스트 자동 생성
- **Step 2 (강제):** 각 체크리스트 항목 완료 증거 첨부 (랜딩 전환율, 인터뷰 노트 등)
- **Step 3 (판정):** 모든 항목 통과 시에만 "Build 허가" 배지 발급
- **부가 기능:** 검증용 랜딩페이지 자동 생성, 인터뷰 질문 템플릿, 경쟁사 분석 가이드

### 1.3 성공 지표 (KPIs)

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| **검증 완료율** | > 70% | 시작된 검증 중 모든 체크리스트 완료 비율 |
| **Build 허가 후 성공률** | > 50% | Build 허가 받은 아이디어 중 런칭 후 100명+ 사용자 달성 |
| **반복 사용률** | > 60% | 첫 검증 후 2번째 아이디어도 검증하는 사용자 비율 |

**Anti-Metric (경계해야 할 지표):**
- 검증 스킵률: 검증 중단하고 바로 빌드하는 사용자 < 20%

## 2. 타겟 사용자

### 2.1 주요 페르소나

**페르소나 1: 반복 실패자 - 준혁 (29세, 풀스택 개발자)**

| 항목 | 내용 |
|------|------|
| 배경 | 3년간 7개 사이드 프로젝트, 모두 실패 (MAU 0-50) |
| 행동 패턴 | 주말에 아이디어 떠오름 → 금요일 밤부터 코딩 시작 → 일요일 저녁 "대충 완성" → 런칭 → 2주 후 방치 |
| 인용 | "이번엔 진짜 될 것 같았는데... 매번 이래" |
| 좌절 포인트 | 시간과 에너지 낭비에 대한 자괴감, "내가 센스가 없나?" |
| 이상적 상태 | 검증 단계를 거쳐 확신을 갖고 빌드 시작 |
| ValidationFirst가 주는 가치 | 흥분 상태에서 브레이크 역할, 검증 없이 시작하면 경고 |

**페르소나 2: AI 코딩 중독자 - 미나 (26세, 주니어 개발자)**

| 항목 | 내용 |
|------|------|
| 배경 | Cursor로 "1시간 만에 SaaS 뼈대 완성" 경험에 매료 |
| 행동 패턴 | 아이디어 → Cursor로 30분 만에 프로토타입 → "우와 이거 되네?" → 런칭 → 실패 → 다음 아이디어 |
| 인용 | "만드는 건 이제 너무 쉬워서... 검증이 더 시간 낭비 같아요" |
| 좌절 포인트 | 빠르게 만들 수 있으니 "검증 시간이 아깝다"는 왜곡된 인식 |
| 이상적 상태 | 빠른 빌드 능력을 검증된 아이디어에만 투입 |
| ValidationFirst가 주는 가치 | "만들기 쉬워졌다고 아무거나 만들면 안 된다" 리마인더 |

**페르소나 3: 신중파 실행 장애 - 동현 (34세, PM 출신 예비 창업자)**

| 항목 | 내용 |
|------|------|
| 배경 | 회사에서 PM으로 검증의 중요성은 알지만, 1인으로 어떻게 하는지 모름 |
| 행동 패턴 | 아이디어 → "검증해야지" → 어떻게? → 계획만 세움 → 실행 안 함 → 3개월 후 다른 사람이 만든 것 발견 |
| 인용 | "검증해야 하는 건 아는데, 1인으로 어떻게 하죠?" |
| 좌절 포인트 | 검증 방법론은 아는데 실행으로 옮기지 못함 (실행 마비) |
| 이상적 상태 | 명확한 체크리스트와 템플릿으로 검증 실행 |
| ValidationFirst가 주는 가치 | "무엇을 해야 하는지" 명확한 가이드 + 진행률 시각화 |

### 2.3 사용자 시나리오

**시나리오 1: 금요일 밤의 충동 억제**

1. 준혁이 금요일 밤 10시, 새 SaaS 아이디어가 떠오름
2. 평소 같으면 바로 터미널 열고 `npx create-next-app` 실행
3. 이번엔 ValidationFirst를 먼저 열고 아이디어 등록
4. 자동 생성된 검증 체크리스트:
   - [ ] 경쟁사 3개 이상 분석 완료
   - [ ] 타겟 고객 5명 인터뷰 완료
   - [ ] 검증용 랜딩페이지 전환율 5% 이상
   - [ ] "이 문제로 돈을 쓴 적 있는가?" 확인
5. "아... 일단 내일 경쟁사부터 찾아보자" → 충동 억제됨
6. 토요일: 경쟁사 분석 결과 이미 3개의 잘 만든 제품 존재
7. No-Go 판정 → "만들었으면 또 시간 낭비할 뻔"

**시나리오 2: AI 코딩 속도의 올바른 활용**

1. 미나가 ValidationFirst로 3주간 아이디어 검증
2. 검증 결과: 전환율 8%, 인터뷰 5/5 긍정, 경쟁사 약점 발견
3. Build 허가 배지 획득
4. 이제 Cursor로 자신 있게 개발 시작
5. 1주 만에 MVP 완성 → 런칭 → 대기자 200명 중 50명 첫 주 가입
6. "검증하고 만드니까 런칭이 무섭지 않았어요"

**시나리오 3: 검증 실행 가이드**

1. 동현이 ValidationFirst에 창업 아이디어 등록
2. 체크리스트와 함께 **실행 가이드** 제공:
   - 경쟁사 분석: "이 키워드로 검색하세요" + 분석 템플릿
   - 인터뷰: 질문 리스트 + 노트 템플릿 + 인터뷰이 찾는 방법
   - 랜딩페이지: 한 줄 요약 입력 → 자동 생성
3. 각 단계별 진행률 시각화 (30% → 60% → 100%)
4. 2주 후 모든 항목 완료, Build 허가
5. "혼자서도 체계적으로 검증할 수 있구나"

## 3. 기능 요구사항

### 3.1 MVP 필수 기능 (3개)

**기능 1: 검증 체크리스트 자동 생성 (핵심)**

| 항목 | 내용 |
|------|------|
| 입력 | 아이디어 한 줄 설명, 타겟 고객, 해결하려는 문제 |
| 출력 | 맞춤형 검증 체크리스트 (5-7개 항목) |
| 체크리스트 예시 | 경쟁사 분석, 고객 인터뷰, 랜딩페이지 테스트, 지불 의사 확인, 시장 규모 추정 |
| 완료 조건 | 각 항목에 증거 첨부 (스크린샷, 노트, 링크 등) |
| 본질적 가치 | **검증 없이 빌드 불가** - 프로세스 강제로 습관 형성 |

**기능 2: 검증용 랜딩페이지 원클릭 생성**

| 항목 | 내용 |
|------|------|
| 입력 | 아이디어 설명 + 가치 제안 + 타겟 고객 |
| 출력 | 호스팅된 랜딩페이지 (idea-name.vfirst.io) |
| 포함 요소 | 헤드라인, 가치 제안, 이메일 수집 폼, 대기자 현황 |
| 분석 기능 | 방문자 수, 전환율, 체류 시간, 스크롤 깊이 |
| 본질적 가치 | **마찰 제거** - 30분 안에 시장 반응 테스트 시작 가능 |

**기능 3: Go/No-Go 판정 대시보드**

| 항목 | 내용 |
|------|------|
| 입력 | 각 체크리스트 항목의 결과 데이터 |
| 출력 | Go / Conditional Go / No-Go / Need More Data 판정 |
| 판정 기준 | 전환율, 인터뷰 긍정률, 경쟁 강도, 시장 규모 등 종합 |
| 설명 제공 | 판정 근거와 개선 제안 (예: "전환율이 낮습니다. 가치 제안 메시지를 수정해보세요") |
| 본질적 가치 | **의사결정 지원** - 데이터 해석 부담 제거, 명확한 다음 단계 제시 |

### 3.2 향후 기능 (Phase 2)

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 인터뷰 가이드 생성 | 아이디어 기반 맞춤 질문 리스트 + 노트 템플릿 | 높음 |
| 경쟁사 자동 분석 | 키워드 기반 경쟁사 검색 + 장단점 요약 | 높음 |
| A/B 테스트 | 가치 제안, 가격, 디자인 변형 테스트 | 중간 |
| 사전 결제 통합 | Stripe Pre-order 연동 (진짜 돈 내는지 검증) | 중간 |
| 커뮤니티 공유 | Reddit, Twitter, HN 자동 포스팅 | 낮음 |
| 검증 성공/실패 사례 라이브러리 | 다른 사용자의 익명화된 검증 사례 열람 | 낮음 |

### 3.3 명시적 제외 (Out of Scope)

| 제외 기능 | 이유 | 대안 |
|-----------|------|------|
| 제품 개발 지원 | 개발 도구 영역 (Cursor, Vercel 등) | Build 허가 후 외부 도구 연동 |
| 본격 마케팅 자동화 | Mailchimp, ConvertKit 영역 | 검증 단계 이메일만 지원 |
| 결제 처리 | Stripe, Gumroad 영역 | Phase 2에서 연동만 지원 |
| CRM 기능 | 고객 관리는 검증 이후 영역 | 외부 CRM 연동 |
| SEO 최적화 | 장기 마케팅 영역 | 검증용 랜딩은 노인덱스 |

### 4.2 기술 스택

**Frontend**
| 기술 | 선택 이유 |
|------|-----------|
| Next.js 14 (App Router) | SSR/SSG 지원, Vercel 최적화 |
| Tailwind CSS + shadcn/ui | 빠른 UI 개발, 일관된 디자인 |
| Zustand | 간단한 상태 관리 |
| Recharts | 분석 대시보드 차트 |
| React Hook Form + Zod | 폼 검증 |

**Backend**
| 기술 | 선택 이유 |
|------|-----------|
| FastAPI (Python) | 빠른 개발, AI 라이브러리 통합 용이 |
| SQLAlchemy | ORM, 마이그레이션 지원 |
| Celery + Redis | 비동기 작업 (랜딩페이지 생성, 이메일 발송) |

| Cloudflare | DNS, 서브도메인, CDN |

    score = 0
    max_score = 100

    # 랜딩 전환율 (30점)

        score += 30

    if positive_ratio >= 0.8:
        score += 30

        score += 20
    elif positive_ratio >= 0.4:
        score += 10

    # 경쟁 강도 (20점)
    if validation_data.competitor_weakness_found:
        score += 20
    elif validation_data.competitor_count < 3:
        score += 15
    elif validation_data.competitor_count < 5:
        score += 10

        score += 20
    elif validation_data.willing_to_pay >= 3:
        score += 15
    elif validation_data.willing_to_pay >= 1:
