# PRD: CodeDebt Journal

> AI가 생성한 코드의 "부채"를 추적하고, 수동 개입이 필요한 지점을 사전에 알려주는 개발자 도구

## 1. 개요 (Overview)

### 1.1 문제 정의 (Problem Statement)

- **누가** 겪는 문제인가?
  - AI 코딩 도구(Copilot, Cursor, Claude Code 등)를 1년 이상 사용해온 시니어 개발자
  - AI 코딩을 팀에 도입한 후 코드베이스가 급격히 커진 테크리드
  - "Vibecoding"으로 빠르게 MVP를 만들었지만, 이제 유지보수 단계에 진입한 창업자

- **무엇이** 문제인가?
  - AI가 생성한 코드는 "즉시 동작"하지만, 시간이 지나면 "왜 이렇게 짜여졌는지" 이해하기 어려움
  - 코드베이스 내에서 "어디가 AI가 만든 부분인지" 구분이 안 됨
  - AI 생성 코드의 품질 편차가 크고, 이것이 기술 부채로 누적됨
  - 수동으로 리팩토링해야 할 부분을 사전에 식별하지 못해 부채가 폭발

- **왜** 문제인가?
  - **Knowledge Collapse**: 개발자가 자기 코드베이스를 이해하지 못하는 상황 발생
  - **유지보수 비용 급증**: AI 코드는 빠르게 만들어졌지만, 고치는 데는 더 오래 걸림
  - **전제 파괴**: "AI 코드는 빠르니까 좋다"는 믿음이 장기적으로는 틀림
  - **시간 폭탄**: 지금은 괜찮아 보이지만, 6개월-1년 후 부채가 한꺼번에 터짐

### 1.2 솔루션 요약 (Solution Summary)

- **핵심 가치 제안**: AI가 생성한 코드를 자동으로 태깅하고, 시간에 따른 부채 누적을 시각화하며, 리팩토링이 필요한 시점에 사전 알림을 제공한다.

### 1.3 성공 지표 (Success Metrics)

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| AI 코드 탐지 정확도 | 90% 이상 | 수동 라벨링 샘플 대비 정확도 |
| 부채 알림 → 리팩토링 전환율 | 30% 이상 | 알림 후 7일 내 해당 파일 수정 여부 |
| 사용자 주간 접속률 | 60% 이상 | 주 1회 이상 대시보드 방문 |
| 월간 이탈률 (Churn) | 8% 미만 | 유료 구독 취소율 |
| "부채 해소 후 만족도" | 4.0/5.0 이상 | 리팩토링 완료 후 사용자 피드백 |

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 MVP 필수 기능 (Must Have)

#### F1: AI Origin Tracker
- **설명:** 커밋, 파일, 함수 단위로 "AI 생성 비율"을 자동으로 탐지하고 태깅
- **사용자 스토리:** As a developer, I want to see which parts of my codebase were AI-generated, so that I can identify areas that might need manual review.
- **수용 기준:**
  - [ ] Git 히스토리 분석으로 AI 도구 사용 흔적 탐지 (커밋 패턴, 코드 스타일)
  - [ ] IDE 플러그인과 연동하여 실시간 AI 코드 생성 이벤트 캡처 (Cursor, Copilot)
  - [ ] 파일별 "AI 생성 비율" 0-100% 표시
  - [ ] 함수/클래스 단위로 AI 생성 여부 마킹
  - [ ] 탐지 결과에 대해 사용자가 수동 보정 가능 ("이건 AI 아님" 표시)

#### F2: Debt Score Dashboard
- **설명:** 시간에 따른 AI 코드 부채 누적을 시각화하고, 코드베이스 건강도 점수 제공
- **사용자 스토리:** As a tech lead, I want to visualize how AI-generated code debt is accumulating over time, so that I can make informed decisions about when to refactor.
- **수용 기준:**
  - [ ] 전체 코드베이스의 "AI Debt Score" 0-100 표시 (낮을수록 건강)
  - [ ] 시간별 추이 그래프 (일/주/월 단위)
  - [ ] 모듈/디렉토리별 부채 히트맵
  - [ ] 부채 점수 산정 기준 투명하게 공개 (AI 비율, 수정 빈도, 복잡도 등)
  - [ ] 경쟁사/업계 벤치마크 비교 (익명화된 집계 데이터)

#### F3: Refactor Alert
- **설명:** 특정 AI 코드가 "수동 검토/리팩토링 필요" 임계치에 도달하면 사전 알림
- **사용자 스토리:** As a developer, I want to be notified before AI-generated code becomes a critical problem, so that I can refactor proactively.
- **수용 기준:**
  - [ ] 부채 점수가 사용자 지정 임계치(기본 70점) 초과 시 알림
  - [ ] 알림에 "왜 위험한지" AI 분석 포함 (예: "이 모듈은 6개월간 수정 없이 AI 비율 85%")
  - [ ] 알림에 "권장 조치" 포함 (예: "테스트 커버리지 추가 권장")
  - [ ] Slack, 이메일, IDE 내 알림 지원
  - [ ] 알림 빈도 조절 가능 (일간/주간 다이제스트)

### 3.2 향후 기능 (Nice to Have)
- **AI Code Explainer**: AI가 생성한 코드에 대해 "왜 이렇게 짜여졌는지" 추론 및 문서화
- **Refactor Assistant**: 부채 높은 코드에 대해 리팩토링 제안 자동 생성
- **Team Analytics**: 팀원별 AI 코드 생성/리팩토링 패턴 분석
- **PR Integration**: PR 리뷰 시 AI 생성 코드 하이라이트
- **Tech Debt Backlog**: 리팩토링 필요 항목을 Jira/Linear 이슈로 자동 생성

### 3.3 명시적 제외 (Out of Scope)
- **코드 생성 기능**: 기존 AI 코딩 도구와 경쟁하지 않음. 추적/분석만.
- **일반 코드 품질 분석**: SonarQube 대체 아님. AI 컨텍스트에 특화.
- **보안 취약점 스캔**: Snyk 등 기존 도구 영역. AI 부채에 집중.
- **테스트 자동 생성**: 범위 밖. 알림 시 테스트 추가 권장만.

- TAM: $500M - $1.5B

- AI 코딩 1년 이상 사용 + 코드베이스 규모 10만 줄 이상: 약 50만 명
- SAM: $50M - $150M

**SOM (Serviceable Obtainable Market)**
- 초기 2년 내 확보 가능: 2,000명
- SOM: $420k ARR

## 부록: Essence Track 관점

### 파괴하는 전제
> "AI 코드는 빠르니까 좋다"

AI 코딩 도구의 마케팅은 모두 "속도"를 강조한다. 10배 빠르게, 80% 자동화. 하지만 아무도 "그 코드를 6개월 후에 유지보수할 때"를 말하지 않는다. 속도에는 대가가 있고, 그 대가는 "부채"라는 형태로 미래로 전가된다. CodeDebt Journal은 이 숨겨진 비용을 가시화한다.

### 방치된 이유
- AI 코딩이 본격화된 것은 2023년 이후, 아직 2-3년밖에 안 됨
- "vibecoding"으로 만든 프로젝트들이 유지보수 단계에 막 진입 중
- 부채가 "터지기 전"이라 아무도 심각하게 안 다룸
- AI 도구 회사들은 "생성"에만 집중, "사후 관리"는 관심 밖

### 비합리적 행동의 증거
- 개발자들이 AI 생성 코드를 리뷰 없이 머지하고 "나중에 보자" 방치
- "TODO: AI generated" 주석 달아놓고 영원히 안 봄
- 버그 나면 그때서야 "이게 AI가 만든 거였네" 깨달음
- Karpathy가 80% AI 코딩한다고 하니 "나도!" 하면서 리스크는 안 봄

### 진정성 요구사항
이 제품은 반드시 **본인이 AI 코딩을 1년 이상 헤비하게 사용해본 사람**이 만들어야 한다. 그렇지 않으면:
- "AI가 만든 코드"의 특유의 패턴을 모름 (과도한 주석, 불필요한 에러 처리 등)
- 부채가 쌓이는 속도감을 체감 못함
- "정말 리팩토링이 필요한 순간"의 기준을 못 잡음
- vibecoding 경험자만 아는 "아, 이건 AI가 한 거구나" 직감이 없음

**권장 사항**: 최소 3개월간 본인 프로젝트를 100% AI 코딩으로 진행해보고, 그 후 유지보수하며 느낀 고통을 기록할 것. 그 기록이 이 제품의 진정한 요구사항 문서가 된다.
